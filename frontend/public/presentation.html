Why each service
13 choices. 13 reasons.
Cloud Armor
DDoS protection · rate limiting
⭐
In plain words
My app is on the internet — anyone can try to flood it with requests to break it. Cloud Armor is the bouncer: it blocks bad actors automatically before they reach anything.
Technical

The moment I exposed my GKE Gateway, I had a public IP. A script sending thousands of requests per second could take down all 3 services at once. Cloud Armor sits in front of the Gateway — same infrastructure Google uses for Search and Gmail. Rate limit: 1000 req/10s/IP → HTTP 429. Attached to all 3 backend services. Traffic filtered at the network layer before reaching any pod.

GKE — 3 independent microservices
songs · playlists · streaming · each scales on its own
⭐
In plain words
Instead of one big app, I split the backend into 3 focused apps. When one gets busy, only that one scales up — the others stay small. More efficient, more resilient.
Technical

Cloud Run would have been simpler. I chose GKE to learn Kubernetes properly. Each service has its own HPA: 2–10 pods at 70% CPU. If streaming spikes, only streaming-service scales. Workload Identity: pods authenticate to GCP without JSON key files — no leak risk, no rotation headache.

songs-service (:3001) — catalog, search, trending, recommendations
playlists-service (:3002) — playlist CRUD
streaming-service (:3003) — play/skip/like events → Pub/Sub
Cloud Pub/Sub → BigQuery
Async analytics pipeline · fire-and-forget
⭐
In plain words
When you press play, the app says "done!" instantly and quietly records it in the background. You never wait. That's Pub/Sub — an invisible background messenger.
Technical

A direct BigQuery write adds 100–500ms to every play event. Instead, streaming-service publishes to Pub/Sub and returns 200 OK immediately. A native BigQuery subscription ingests events automatically — zero consumer code written. Messages persist 7 days if BigQuery is unavailable. 1M+ rows (by script) power trending and recommendations.

Cloud Firestore
Operational database · songs, playlists, likes
⭐
In plain words
The main database — stores the song catalog, your playlists, and liked songs. All 3 apps share it. It runs and scales itself, no maintenance needed.
Technical

Firestore over Cloud SQL: Jamendo metadata is inconsistent across tracks — flexible schema is a genuine advantage. No connection pool, native Firebase Auth integration, security rules enforced at DB level. Intentional separation: Firestore for operational reads/writes, BigQuery for analytical queries.

Firebase Authentication
User identity · JWT on every request
⭐
In plain words
When you log in, you get a digital pass (JWT token). Every action your browser takes, it shows that pass to the backend. Without it, the backend refuses to respond.
Technical

Firebase Auth issues a signed JWT on login. Every API route on all 3 services verifies it server-side via Firebase Admin SDK. Chosen for native Firestore security rules integration — same identity enforced at API and database layers. 51 test users created by script.

Firebase Hosting + Cloud CDN
Global frontend delivery
⭐
In plain words
When you type playwavely.web.app, the app loads from the server closest to you — not from a single machine in France. Paris, Tokyo, New York — same fast experience.
Technical

Angular SPA on Firebase Hosting, automatically backed by Cloud CDN. Chosen because Cloud Build deploys directly via a Firebase CLI token stored in Secret Manager — seamless CD pipeline. Zero server management, automatic SSL, global edge caching.

Cloud Storage
1000 MP3 files · SPA reads directly · no backend proxy
⭐
In plain words
The music files are stored in Google's cloud. When you press play, your browser fetches the audio directly — no middleman app in between. Faster, simpler, cheaper.
Technical

The SPA streams MP3s directly from Cloud Storage URLs — bypassing all 3 microservices entirely. Audio through the API would add latency, waste pod resources, and introduce a failure point. Direct object storage is the right architecture for media delivery. Files sourced from Jamendo by script (Creative Commons).

Secret Manager
API keys · Firebase token · Angular env — never in git
⭐
In plain words
API keys must never be public. Secret Manager is a secure vault — keys stay encrypted there, and only the build system can fetch them when needed.
Technical

The Angular environment.ts contains Firebase API keys — committing it to GitHub exposes them. Cloud Build fetches it from Secret Manager at build time and injects it. Never in the repository. Same for the Firebase CLI deployment token.

IAM + Workload Identity
GKE pod authentication · no key files anywhere
⭐
In plain words
The 3 apps need permission to access the database. Instead of a password file (which can be stolen), Workload Identity gives each app a digital ID that Google recognises automatically — nothing to steal.
Technical

Without Workload Identity, each pod needs a JSON key file — files don't expire, can leak in images or logs, painful to rotate. Workload Identity links the Kubernetes service account to a GCP IAM service account via the metadata server. Zero credentials stored anywhere in the cluster.

Cloud Build
Continuous deployment · git push → live in ~10 min
⭐
In plain words
Every time I push code to GitHub, the app updates itself automatically in ~10 minutes. No manual work needed.
Technical

Trigger fires on every push to main: builds 3 Docker images → Artifact Registry → kubectl rollout restart → fetches environment.ts from Secret Manager → ng build → firebase deploy. Backend + frontend in one cloudbuild.yaml. CD, not CI — no automated tests yet.

Artifact Registry
Docker image storage · same region as GKE
⭐
In plain words
When Cloud Build packages the 3 apps, it creates containers — like boxes with everything needed to run. Artifact Registry is the warehouse where those boxes live before deployment.
Technical

Images stored in europe-west1 — same region as the GKE cluster to avoid cross-region egress costs and reduce pull latency. Replaces Container Registry with finer IAM control per repository.

GKE Gateway — L7 Load Balancer
Single entry point · path routing · Cloud Armor attachment
⭐
In plain words
3 backend apps, one address. The Gateway reads the URL and sends each request to the right app — like a receptionist directing visitors in a building.
Technical

gke-l7-global-external-managed with HTTPRoute. Chosen over Ingress for the modern GKE-native approach and Cloud Armor attachment directly to backend services. SSL via Certificate Manager. /songs/* · /playlists/* · /streaming/*.
